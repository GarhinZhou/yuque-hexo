## Binary Patch
注意！！！

在打补丁加汇编指令的时候不能乱动栈！！！要么全乱了要么就是调用函数的时候栈不平衡

### IDA保存patch后的程序
![](/images/d08339850baf15292a9730cf9838d067.png)

保存的时候遇到了Patching canceled...的问题愣是导出不了，

搜了一下发现是IDAPro依赖的python版本太低，缺了一些东西

解决方法：

先用pip安装PyQt5

```bash
pip install PyQt5
```

安装完之后到安装python的目录里依次进入`Lib/site-packages/PyQt5`

在这里找到安装好的PyQt5，把里面的`sip.cp311-win_amd64.pyd`文件复制

然后到IDAPro的目录接着进入`python/3/PyQt5`，替换掉里面的`sip.pyd`文件

我这里直接没有这个文件，直接放进去就行了

然后重启IDAPro就可以正常导出了

## 格式化字符串修复
### 改函数参数
把`printf(buf)`改成`printf("%s",buf)`（程序如果本来有%s的话）

或者如果程序没PIE的话，可以把%s直接写到`eh_frame`段上

### 换调用函数
跟栈溢出那边做法一样

但是注意如果把printf换成puts的话会在字符串后面加上换行符，一些严格的check脚本可能不过

### 动态链接换函数
可以从符号表下手，跟ret2dlresolve相似，相当于NO RETRO情况直接改dynstr

如果本来函数名就比要改的函数名长，可以直接在名字上改

比如把printf改成puts函数，不需要动dynsym里面定位dynstr的偏移，直接把dynstr改成对应的函数名（记得字符串以\x00结尾）

![](/images/6913d7536d8154c1a59c00fa00c536cc.png)

如果本来函数名就比较短，就在`eh_frame`段写上函数名字符串，再把dynsym当中定位dynstr的偏移patch成`eh_frame`上面字符串相对的偏移

![](/images/1dbb41eb3bcd1463504d74f646d9ee49.png)

## 通防
### 缩短alarm时间
因为现在的 PWN 题运行过程动不动就有个 5s 以上（比如说要打 IO 爆破），而 checker 的运行只有短短 1~5 秒，所以可以从alarm下手

alarm 函数会在时间到了以后强制结束进程，那么可以 patch 程序最开始的 alarm(60) 为 alarm(3)，这样就可能让 checker 通过而攻击脚本不通过，从而通过 check 逻辑

### 沙盒工具
github：https://github.com/TTY-flag/evilPatcher/tree/main

```bash
evilPatcher.py elfFile sandboxFile.asm
```

沙箱规则放在sandbox.asm，例如：

```plain
A = sys_number
A >= 0x40000000 ? dead : next
A == execve ? dead : next
A == open ? dead : next
return ALLOW
dead:
return KILL
```

原理：

也是在`eh_frame`打补丁，但是改了一下`_libc_start_main`函数的参数，让它先调用在`eh_frame`上写的一段开启沙箱的汇编指令，再跳回到原来的main函数

这里的汇编指令实际上是通过pwntools生成的

实际上binary是没有程序定义的函数的符号的，只是根据binary开头的EntryPoint来分析出start函数里面的_libc_start_main函数调用的是main函数来改的名字，第一眼还以为是重写了一遍main函数

![](/images/af71d4f11a8a67597a2cdd0851d1f494.png)

![](/images/47a59e3027ffae1bba7baf55b5b44057.png)

### 编译
这个步骤在本机进行比较方便，不知道服务器上面有没有对应的库和make，而且也是静态编译，可以直接从本机把编译之后的elf直接传到服务器上用

把src源码文件夹和makefile文件下下来，放在同一个目录下

然后在makefile里面配置编译选项：（CATCH模式就只用改红色背景的部分，必须用绝对路径）

LOG_PATH是服务器的目录，要确保有读写权限

```plain
# Author: i0gan
# Do    : For compile waf program
# Date  : 2020-12-15

GCC    := gcc
CFLAGS := -O2 --static
RM     := rm -rf

# configure
# log path
LOG_METHOD  := OPEN
LOG_PATH    := /home/pwner/WAF/log
ARCH        := 64

# Just used in FORWARD mode
FORWARD_IP   := 127.0.0.1
FORWARD_PORT := 20000

# src
TEST_PWN_SRC := src/test_pwn.c
HEX_SRC      := src/hex.c
LOGGER_SRC   := src/logger.c
WAF_SRC      := src/waf.c

all : catch i0gan forward forward_multi test_pwn

test_pwn :
    @mkdir $(LOG_PATH)
    $(GCC) $(TEST_PWN_SRC) -o $@
    @cp $@ $(LOG_PATH)/pwn
    @cp hosts.txt $(LOG_PATH)/

clean :
    $(RM) catch i0gan forward forward_multi test_pwn $(LOG_PATH)

catch :
    $(GCC) $(CFLAGS) $(WAF_SRC) $(LOGGER_SRC) -DRUN_MODE=CATCH -DLOG_METHOD=$(LOG_METHOD) -DLOG_PATH=\"$(LOG_PATH)\" -DARCH=$(ARCH) -o $@

i0gan :
    $(GCC) $(CFLAGS) $(WAF_SRC) $(LOGGER_SRC) -DRUN_MODE=I0GAN -DLOG_METHOD=$(LOG_METHOD) -DLOG_PATH=\"$(LOG_PATH)\" -DARCH=$(ARCH) -o $@

forward : 
    $(GCC) $(CFLAGS) $(WAF_SRC) $(LOGGER_SRC) -DRUN_MODE=FORWARD -DLOG_METHOD=$(LOG_METHOD) -DLOG_PATH=\"$(LOG_PATH)\" -DARCH=$(ARCH) -DFORWARD_IP=\"$(FORWARD_IP)\" -DFORWARD_PORT=$(FORWARD_PORT) -o $@

forward_multi : 
    $(GCC) $(CFLAGS) $(WAF_SRC) $(LOGGER_SRC) -DRUN_MODE=FORWARD_MULTI -DLOG_METHOD=$(LOG_METHOD) -DLOG_PATH=\"$(LOG_PATH)\" -DARCH=$(ARCH) -o $@
```

这里里面的LOG_PATH既是放log日志文件的地方，实际上也是放题目的elf的地方

（实际题目的elf要改名pwn，记得给可执行权限）

在makefile同一目录下（CATCH模式就编译catch）：

```bash
make catch
```

64位的编译没有任何问题，但是32位可能会报错，因为gcc默认是以64位编译的

### 部署
这部分就在服务器上进行了

然后就在本机当前目录得到了一个叫catch的elf文件，上传到服务器

把题目的elf复制到定好的LOG_PATH：

```bash
cp /pwn/pwn_elf LOG_PATH
```

最后用前面编译的catch替换原本题目的elf文件（名字也要换，用mv指令就直接替换然后改成原本题目的elf的名字了）：

```bash
mv /path/catch /pwn/pwn_elf
```

实际上编译出来的catch是根据makefile里面的LOG_PATH找到题目文件来执行，从而实现监听和转发流量

重点：

1. makefile当中的LOG_PATH：路径要有读写权限，原本题目的elf存放在这个路径下，并且改名pwn
2. makefile当中的ARCH：按照题目elf是32位还是64位来选择
3. 编译后的catch文件替换真正执行的elf
